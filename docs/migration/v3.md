# Migrating to v3.0.0

v3.0.0 removes every non-isolated code path from the library. There is no
longer a shared, module-level "active chain" singleton anywhere:
`multichain.create(chainName)` now always returns a chain-lib whose classes
close over their own private `Networks` instance and hash registry.

## What changed

- **`create()` is now always isolated.** Previously, `create()` mutated a
  shared singleton (`lib/_context.js`) and a shared hash-algorithm registry,
  so two chain-libs obtained via `create()` (for the same chain or different
  ones) were not independent — the last `create()` call could affect a
  chain-lib returned earlier. That singleton is gone. `create()`'s call
  signature and return shape are unchanged; only its internal behavior is
  fixed.
- **`createIsolated()` has been removed** — it's no longer needed, since
  `create()` now does what `createIsolated()` used to do. Replace
  `multichain.createIsolated(chainName)` with `multichain.create(chainName)`.
- **`multichain.registerAlgorithm(name, fn)` and `multichain.algorithms()`
  have been removed.** These operated on the old shared, global hash
  registry. Register custom algorithms per chain instead, in the config
  passed to `registerChain()`:

  ```javascript
  multichain.registerChain('mychain', {
    livenet: { /* ... */ },
    testnet: { /* ... */ },
    algorithms: {
      myalgo: (buf) => /* ... */,
    },
  });
  ```

  To inspect or use a chain's registered algorithms at runtime, use its
  chain-lib's `crypto.Hash` (e.g. `mine.crypto.Hash.list()`,
  `mine.crypto.Hash.forNetwork(buf, mine.Networks.livenet)`), which is a
  registry private to that chain. A standalone registry, independent of any
  chain, can still be created with `multichain.createHashRegistry()`.
- **`lib/_context.js` has been deleted.**

## How to migrate

If you were already using `createIsolated()`:

```diff
- const chainLib = multichain.createIsolated(chainName);
+ const chainLib = multichain.create(chainName);
```

If you were using `create()` directly, no code changes are required — the
call signature and the shape of the returned chain-lib are identical. Only
its guarantees improved: it is now safe to hold onto multiple chain-libs
(for the same or different chains) and use them concurrently, including
across `await` boundaries, without one corrupting another.

If you were using `multichain.registerAlgorithm()` / `multichain.algorithms()`:

```diff
- multichain.registerAlgorithm('myalgo', fn);
- multichain.algorithms(); // ['x11', 'myalgo', ...]
+ multichain.registerChain('mychain', {
+   livenet: { /* ... */ },
+   testnet: { /* ... */ },
+   algorithms: { myalgo: fn },
+ });
+ const mine = multichain.create('mychain');
+ mine.crypto.Hash.list(); // ['x11', 'myalgo', ...]
```

## Previously-documented limitations, now resolved

An earlier draft of the isolation work (additive, never released) documented
several known limitations of the isolated path. All of them are resolved in
this release — there is no remaining non-isolated code path in the library:

- `Transaction#from(utxo)` with a plain address string, `Transaction#to()`,
  and `Transaction#change()` are fully isolated.
- `ProRegTxPayload#toJSON()` / constructor's `scriptPayout` address handling
  is fully isolated.
- `ProRegTxPayload#verifySignature()` automatically uses its own chain's
  `PublicKey` class.
- `Mnemonic#toHDPrivateKey()` uses its own chain's `HDPrivateKey` class.
- The entire `Transaction` input/output subsystem (`Input`, `Output`,
  `UnspentOutput`, `TransactionSignature`, and the `PublicKeyHash`/
  `PublicKey`/`MultiSig`/`MultiSigScriptHash` input types) is isolated per
  chain.
- `Interpreter` (the script evaluator) is isolated per chain.

## Low-level crypto primitives

A few low-level, chain-agnostic crypto helpers now require an explicit,
chain-bound class to be passed in where they used to fall back to a
legacy default:

- `ECDSA.prototype.toPublicKey(PublicKeyClass)` — `PublicKeyClass` is now
  required (previously defaulted to a legacy `PublicKey`).
- `Signer.verifyHashSignature(hash, signature, publicKeyId, PublicKeyClass)`
  — `PublicKeyClass` is now required.
- `Signer.sign(data, privateKey)` / `Signer.signHash(hash, privateKey)` now
  require `privateKey` to already be an instance of a chain's `PrivateKey`
  class (previously accepted a raw string and constructed a legacy
  `PrivateKey` from it).

These are only relevant if you call `lib/crypto/ecdsa.js` or
`lib/crypto/signer.js` directly; normal usage through a chain-lib's
`Message`, `Transaction`, or `ProRegTxPayload` is unaffected.
